          # 1. 移动端适配的难点

          1. 设备像素比dpr不一致；
          2. webview版本较老，不支持新特性。
          3. 想要在不同dpr的屏幕下，区块/文字/边框等视觉效果相同。

          说人话就是，难点是跟随产品经理的需求自己跑来的，即产品经理想让同一网页在不同缩放比率的屏幕上显示相同的效果。

          例如：<div>块的宽度，在不同屏幕的百分比相同；文字和边框，在不同屏幕下肉眼看起来都是16px和1px（在dpr=2的情况下，实际是32px和2px）。

          # 2. 理想的适配方案

          也就是利用vw单位适配：

          1. 在html文件内的<head>内添加<meta name="viewport" content="width=device-width, initial-scale=1.0, user-scalable=no">；
          2. 对html文件内的<body>内的元素的长度和间距等属性使用VW作为单位；
          3. 对html文件内的文字的大小和元素的边框宽度等属性使用px作为单位。

          如下例所示：

          <!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0, user-scalable=no">
    <title>Document</title>
</head>
<body>
    <style>
        #block {
            max-width: 500px;
            width: 50vw;
            height: 50vw;
            font-size: 14px;
            color: white;
            background-color: rgba(0, 0, 0, .8);
            border: 2px solid red ;
        }
    </style>
    <div id="block">
        Destroy All Humans!
    </div>
</body>
</html>

          这样，因为<meta name="viewport">，使页面所有元素（包含文字）可以被用户通过手势全局缩放；又因为content="width=device-width, initial-scale=1.0"，使页面的文字能被浏览器自动放大为物理的42px、肉眼的14px，而我们则不用单独编写js脚本进行控制了。

          幸运的是，现代浏览器大都已经实现了这些特性。对于一些兼容问题，可以通过配合使用less/sass/webpack等相关工具进行解决。

          # 3. 关于lib-flexible适配方案

          库的地址：amfe/lib-flexible: 可伸缩布局方案

          官方很久以前就做了说明：

          由于viewport单位得到众多浏览器的兼容，lib-flexible这个过渡方案已经可以放弃使用，不管是现在的版本还是以前的版本，都存有一定的问题。建议大家开始使用viewport来替代此方案。vw的兼容方案可以参阅《如何在Vue项目中使用vw实现移动端适配》一文。

          虽然我们不应在实际项目中使用lib-flexible这个方案，但是了解一下该方案的原理还是有必要的：

          1. 在n年以前，vw/vh等viewport的单位存在兼容问题，于是淘宝工程师想到了用rem单位模拟vw单位；

          2. 假设可视窗口宽度为375px,设计稿宽度为750px，设计稿内某div元素宽度240px，dpr为1，如果使用vw作为单位：

            • (1) 100vw === 375px；
            • (2) 240px/750px = x/100vw，获得x=32vw；
            • (3) 即在375px的视口中，该元素宽度占比32%（即120px）；

          3. 现在要用rem模拟vw，先设定100rem === 100vw：

            • (1) 那么 100rem === 375px;
            • (2) 1rem === 1vw === 3.75px;
            • (3) 所以应该有<html style="font-size: 3.75px;">；
            • (4) 上面提到的div元素，用rem表示为 32rem；

          4. 但是浏览器支持的font-size最小为12px或10px，于是更改设定为 10rem === 100vw：

            • (1) 1rem === 10vw === 37.5px;
            • (2) 于是<html style="font-size: 37.5px;">
            • (3) 然后那个div宽度，用rem表示就应该为 3.2rem（32vw === 3.2rem）；

          5. 如果屏幕为高分辨率，假设dpr=3，就要修改<meta name="viewport" content="...">进行缩放了：

            • (1) 视口为375px，那么在这375px的狭小宽度内，实际显示的宽度是 375px * 3 = 1125px；
            • (2) 对实际宽度进行缩小，缩小率 scale = 1/dpr（即1/3， 0.333333）；那么设定<meta name="viewport" content="initial-scale=0.333333">
            • (3) 10rem === 100vw === 1125px;所以 1rem === 112.5px;
            • (4) 于是<html style="font-size: 112.5px;">
            • (5) 然后那个占比32%的div，在1125px中实际宽度为 112.5px * 3.2rem = 360px（也就是 1125px * 0.32 = 360px）；

          6. 由以上我们知道，用rem模拟vw：

            • (1) 根元素html的font-size（也就是1rem）应该动态计算；10rem === 100vw === 视口宽度 * dpr；
            • (2) 设定<meta name="viewport" content="initial-scale=缩放率">；缩放率 scale = 1/dpr;
            • (3) 设计稿无论是640px、是750px还是任意宽度，只要得到目标元素在稿内的宽度占比即可；若占比25%，即为25vw，那么 width/10rem = 25vw/100vw，求得写css中的 width: 2.5rem;

          7. 注意事项：

            • (1) 文字的话，不应该用rem作为单位，应仍然使用px（原因是防止出现类似 13.2333px 这种奇葩的字体大小）；
            • (2) 由于动态设定了根元素html的font-size，所以会改变整个页面的字体的默认大小，甚至可能是几百px；
            • (3) 为了消除影响，于是动态设定 body 元素的默认 font-size 为 (12 * dpr)px;
            • (4) 其中乘以 dpr 是为了在不同分辨率的屏幕下，消除 meta scale 缩放的影响，使文字在不同分辨率下肉眼可见的大小一致。
