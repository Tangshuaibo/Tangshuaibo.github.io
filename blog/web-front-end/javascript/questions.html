做个减法：

  • GOF原书未提及但菜鸟教程提及的设计模式：
    • 过滤器模式
    • 空对象模式
    • MVC模式
    • 业务代表模式
    • 组合实体模式
    • 数据访问对象模式
    • 前端控制器模式
    • 拦截过滤器模式
    • 服务定位器模式
    • 传输对象模式
  • 建造者模式扩展了工厂模式的工厂对产品的自定义能力：
    • 当一个类的构造函数参数个数超过4个，而且这些参数有些是可选的参数，考虑使用构造者模式。
  • 原型模式在运行时，复制带有状态的实例对象为新对象。
  • 适配器模式用于封装不兼容的接口。
  • 桥接模式保证对外可调用的接口改变时，无需改变被调用的接口。
  • 组合模式以链表/多叉树递归遍历的方式连接各个对象节点。
  • 装饰器模式在不改动旧对象的基础上，建立调用并增强旧对象的新对象。
  • 外观模式提供了子系统/模块的直接调用，外观类有时可以作为适配器模式的适配器类。
  • 享元模式提供了单例集合的多路选择器。
  • 代理模式为对象访问前的操作提供了一个中间层平台。
  • 责任链模式常与组合模式一起使用，javascript事件的捕获和冒泡机制即由责任链模式实现。
  • 命令模式将一个或一组操作请求封装为一个命令对象，以被调用：
    • 可作为过程语言中回调函数机制的一个面向对象的替代品；
    • 支持命令的取消与重做，可用于原子操作与事务；
    • 可维护命令执行历史列表，并保存命令执行时的历史状态，以便恢复历史状态；
    • 宏命令（组合模式）/历史命令保存（原型模式）/历史状态保存（备忘录模式）。
  • 解释器模式用于构建解释器进行词法和语法分析，将语言中句子表示为抽象语法树（草……去重学编译原理以配合理解吧）。
  • 迭代器模式将定义在聚合对象（如组合模式对象）内的遍历查找等操作独立出来：
    • 应用领域较多，去看GOF书内总结，不能再精简了。
  • 中介者模式独立出多个对象间复杂的的引用或通信关系，中介者对象接收转发同事对象发出的信息给其它同事对象：
    • 一些网络文章表示：跨线程或跨进程的通信，常用中介者模式（如android的binder、微信小程序的渲染层与逻辑层间通信）；
    • 一些网络文章表示：观察者模式是单向发布信息，而中介者模式是双向通信，中介者模式可由观察者模式实现消息分发或转发；
    • GOF原书表示：观察者模式和中介者模式是两种相互竞争的模式，原因在于观察者模式将通信分布，而中介者模式将通信封装；
    • 个人认为两者最明显的区别在于，观察者模式将观察者对象列表维护在被观察者对象内部，而中介者模式将通信能力完全托付给中介者对象。
  • 备忘录模式可用于增强原型模式，保存和恢复对象某时刻的内部状态；备忘录列表可用迭代器模式遍历。
  • 关于观察者模式与发布订阅模式的区别（GOF原书并未区别两者）：
    • DOM事件监听的实现基于发布订阅模式；
    • 观察者模式：需要被观察对象维护观察者列表；
    • 发布订阅模式：事件中心不维护订阅者列表，而维护一系列事件列表与事件触发的动作列表；
    • 即区别在于：一个维护的是观察者队列，另一个维护的是事件触发的动作队列；
    • 另有一种说法，将发布订阅模式视为观察者模式的别名，而事件监听机制（event-listener）是观察者模式的子集。
  • 状态模式与策略模式非常类似，不同在于策略模式由输入信息决定策略后不会自动切换策略，而状态模式根据输入信息执行完一些操作后自动切换状态（有限状态机）。
  • 模板方法模式在保证子类继承抽象类时，抽象父类的某些方法无法被子类重写。
  • 访问者模式在不变更被访问对象类结构的基础上，通过固定接口引入第三方访问者对象让其引用自身或自身某元素，达到借助第三方对象扩充自身能力的效果：
    • 表现出了双分派的能力（但被访问者使用内部迭代器，则是单分派）；
    • 对被访问者的遍历，可以是被访问者的内部迭代器、访问者对象内的迭代器以及独立的外部迭代器；
    • 与装饰器模式相比，可以动态切换访问者，动态地按需增强不同能力；
    • 访问者可作为解释器于解释器模式。
